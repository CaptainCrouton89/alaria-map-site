Design review · Codex of Alaria

Why the codex feels bland — and the redesign that fixes it

The codex has a strong aesthetic and a rich relational dataset, but the front door is a flat text list and the entity page buries its best parts. This review proposes a "doorway" landing, a visible mood system, and a tiered image strategy. The mock frames below use the real dark-parchment theme; the lighter chrome is this document.

Problem 1 · The landing is a database dumpEvery row looks identical — same seal, same type size, same muted blurb. A five-level weight hierarchy exists in the data but the list throws it away. A footnote looks as important as an entire plane of reality.
Problem 2 · The front door hides the mapThe tagline promises "travel by its connections," yet the landing shows zero connections, zero map, zero spatial context — on a worldbuilding site whose richest asset is a world map.
Problem 3 · A whole mood system is invisible9 atmospheres (sacred / cursed / dangerous / ancient…) collapse to a faint tint on a 40px seal. A cursed ruin and a sacred grove look identical in a list.
Problem 4 · The blurb is written but never shown on the entity pageYou wrote a sharp one-liner for every entity. It's used in search, then discarded. The entity page jumps title→prose with no hook. Free win.
Locked decisions (from the consult) Audience: both — immersive doorway, search always one keystroke away.  ·  Map ↔ codex: co-equal — landing leads with the world but stands alone via Begin / Browse / Wander.
Update · images are now abundant (~3 per entity)~1,000 free generations/day makes 3 images per entity realistic. This flips the strategy: images stop being scarce "polish" and become a primary design material. The question is no longer which entities get art — it's what role each image plays, and how to keep ~11,000 images coherent. The style lock matters more now, not less. See "Image roles" below.

Your question, re-answered for abundance

Three images per entity — a role system, not three random pictures

With ~1,000 free generations/day, every entity can have 3 images. The discipline shifts from "which entities get art" to "what role each image plays." Three fixed roles, generated from one locked style + per-type framing. Same id + role → same seed → stable image.

Role 1 — Hero (3:1)
Wide establishing banner. Masthead of the entity page; featured slot on the landing. Place→vista, person→portrait, faction→heraldry.
Role 2 — Portrait (1:1)
The "face." Search rows, cards, map-pin popovers, infobox header. Every entry gets one — the workhorse image.
Role 3 — Scene (4:3)
A moment, not a portrait — the deed, the harbor, the rite, the habitat. Embedded mid-prose as an illustrated figure with caption.
The locked style — matters MORE at volume (~11,000 images)Aged hand-illustrated folio plate · sepia & muted earth palette · ink-and-wash with restrained gouache · soft vignette, faintly foxed paper · no text, no border, no modern elements · natural-history engraving, not vivid digital fantasy. Prompt = [locked style] + [role framing] + [subject framing for entityType] + [atmosphere palette modifier] + [per-entity identity descriptor] + [name + key nouns]. You can't hand-check 11k images — the template is the quality control.
Per-entity identity lock — so the 3 images cohereThree images of Gaea must look like the same Gaea. Derive one appearance descriptor per entity (from its prose/tags), store it, and reuse it across all three roles — vary only role + framing, keep a stable per-entity seed. Without this, abundance produces three strangers.

The new risks abundance introduces — plan for themSlop at scale: 11k cheap gens look like 11k artists unless the style lock holds — enforce it in the template, not by hand. Duds: volume produces garbage (extra limbs, garbled text); keep the generative sigil + map crop as automatic fallbacks and a deterministic "regenerate just the bad ones" loop. Performance: 11k images need webp/avif, responsive srcset, lazy loading, and a CDN — reuse the R2 bucket you already run for map tiles.
Sequencing still holdsShip the structural wins first (they stand without art and give graceful no-image states). Then run the pipeline: lock the style on ~20 hand-checked samples → generate portraits first (the workhorse, every entry) → then heroes → then scenes. Each role backfills as progressive enhancement.
